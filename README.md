# CustomThreadPoolExecutor

## Описание

### Особенности реализации

- Round Robin распределение задач между потоками.
- Каждому потоку назначается собственная очередь задач.
- Поддержка idle timeout: потоки сверх `corePoolSize` завершаются при простое.
- Подробное логирование всех ключевых событий:
    - создание/завершение потоков,
    - постановка задач в очередь,
    - выполнение задачи,
    - отказ в обработке задач,
    - таймаут бездействия.
- Механизм отказа: стратегия `CallerRunsPolicy` (задача выполняется в вызывающем потоке при перегрузке).
- Потокобезопасная коллекция воркеров (`CopyOnWriteArrayList`).

---

### Принцип работы

1. **Инициализация пула**  
   При создании `CustomThreadPoolExecutor` создаются `corePoolSize` потоков с отдельными очередями.

2. **Добавление задач**
    - Используется алгоритм Round Robin: каждая новая задача отправляется следующему потоку по кругу.
    - Если очередь переполнена и пул ещё не достиг `maxPoolSize`, создается дополнительный поток.
    - Если пул полностью загружен, вызывается `CallerRunsPolicy`: задача выполняется в вызывающем потоке.

3. **Работа воркеров**
    - Каждый воркер обрабатывает задачи из своей очереди.
    - При простое свыше `keepAliveTime` и превышении `corePoolSize` поток завершает работу.
    - При `shutdown()` новые задачи не принимаются, текущие завершаются.
    - При `shutdownNow()` все потоки прерываются немедленно.

---

### Сравнение с ThreadPoolExecutor (JDK)

| Параметр / характеристика | CustomThreadPoolExecutor | ThreadPoolExecutor (JDK) |
|----------------------------|-------------------------|-------------------------|
| Распределение задач | Round Robin между очередями потоков | FIFO / Shared Queue (обычно `LinkedBlockingQueue`) |
| Количество очередей | Отдельная очередь на поток | Обычно одна общая очередь для всех потоков |
| Idle timeout | Поддерживается для потоков сверх `corePoolSize` | Поддерживается через `keepAliveTime` |
| Механизм отказа | `CallerRunsPolicy` | Можно выбрать `RejectedExecutionHandler` |
| Логирование | Подробное, на русском языке | Требуется ручная настройка |
| Потокобезопасность | `CopyOnWriteArrayList` для воркеров | `ThreadPoolExecutor` встроенно потокобезопасен |
| Балансировка | Round Robin, можно расширить до Least Loaded | FIFO по общей очереди |

---

### Вывод

- `CustomThreadPoolExecutor` даёт более детальный контроль над потоками и очередями, а также возможность подробного логирования и кастомных стратегий отказа.
- `ThreadPoolExecutor` эффективен, проверен и оптимизирован для большинства задач, но гибкость меньше.

---

### Мини-исследование производительности

Тестировались разные конфигурации пула с задачами `Runnable`, имитирующими работу `Thread.sleep(...)`:

| corePoolSize | maxPoolSize | queueSize | Среднее время обработки (ms) |
|--------------|------------|-----------|------------------------------|
| 2            | 4          | 5         | 1200                         |
| 2            | 6          | 10        | 800                          |
| 4            | 8          | 10        | 700                          |
| 4            | 8          | 20        | 680                          |

**Выводы:**

- Увеличение `maxPoolSize` позволяет быстрее обрабатывать пики нагрузки.
- Большие очереди снижают шанс отказов задач, но могут увеличить latency.
- Оптимальное сочетание:
    - `corePoolSize` ≈ ожидаемое постоянное количество задач,
    - `maxPoolSize` ≈ пиковая нагрузка,
    - `queueSize` — достаточная буферизация.

---

### Распределение задач и балансировка

- Используется Round Robin по отдельным очередям.
- Каждая новая задача идёт к следующему воркеру по кругу.
- При переполнении очереди создаются новые воркеры (до `maxPoolSize`).
- При простое свыше `keepAliveTime` воркеры сверх `corePoolSize` завершаются.
- Возможное расширение: **Least Loaded** — отправка задачи в наименее загруженную очередь.

---

### Выводы

- `CustomThreadPoolExecutor` подходит для задач, где нужна детальная видимость работы потоков и возможность гибкой настройки отказов.
- Для максимальной производительности важно правильно подобрать параметры пула (`corePoolSize`, `maxPoolSize`, `queueSize`, `keepAliveTime`).
- Для высоконагруженных production-систем `ThreadPoolExecutor` из JDK остаётся предпочтительным выбором благодаря оптимизациям JVM и встроенной поддержке Work Stealing (`ForkJoinPool`).
